2. KEY DIFFERENCES BETWEEN CRA AND VITE
================================================================================

DIFFERENCE #1: Entry Point Location
------------------------------------
CRA:  index.html is in public/ folder
Vite: index.html is in ROOT directory â˜… MAJOR DIFFERENCE!

DIFFERENCE #2: File Extensions
-------------------------------
CRA:  Uses .js files (App.js, index.js)
Vite: Uses .jsx files (App.jsx, main.jsx) for components

DIFFERENCE #3: Entry JavaScript File
-------------------------------------
CRA:  src/index.js
Vite: src/main.jsx

DIFFERENCE #4: Configuration
-----------------------------
CRA:  Hidden configuration (uses react-scripts)
      Need to "eject" to customize (irreversible!)
Vite: Exposed vite.config.js - easy customization

DIFFERENCE #5: Public Assets
-----------------------------
CRA:  More files by default (PWA manifests, multiple logo sizes)
Vite: Minimal setup (just vite.svg)

DIFFERENCE #6: Testing & Performance Tools
-------------------------------------------
CRA:  Includes setupTests.js, App.test.js, reportWebVitals.js by default
Vite: No testing setup by default (you add what you need)

3. FILE & DIRECTORY SIGNIFICANCE
================================================================================

ROOT LEVEL FILES
----------------
File: package.json
Purpose: 
  - Lists all project dependencies
  - Contains scripts (start, build, test, etc.)
  - Project metadata (name, version, description)
  - Example: "react": "^18.2.0", "react-dom": "^18.2.0"

File: package-lock.json
Purpose:
  - Locks exact versions of all dependencies
  - Ensures consistent installs across different machines
  - Auto-generated, don't edit manually

File: .gitignore
Purpose:
  - Tells Git which files/folders to ignore
  - Common entries: node_modules/, build/, .env.local
  - Prevents large files from being committed

File: README.md
Purpose:
  - Project documentation
  - Setup instructions
  - How to run, build, and deploy

File: vite.config.js (VITE ONLY)
Purpose:
  - Vite-specific configuration
  - Configure plugins, server settings, build options
  - Proxy settings, path aliases

File: index.html (VITE ONLY - in root)
Purpose:
  - Main HTML template
  - Includes <script> reference to main.jsx
  - Contains <div id="root"></div>


DIRECTORY: node_modules/
-------------------------
Purpose:
  - Contains ALL installed npm packages
  - Generated by "npm install" or "npm i"
  - Can be HUGE (100,000+ files)
  
IMPORTANT:
  âœ— NEVER commit to Git
  âœ— NEVER edit files inside
  âœ“ Delete and reinstall if issues arise


DIRECTORY: public/
------------------
Purpose: Static assets served AS-IS without processing

File: index.html (CRA ONLY)
  - Main HTML template
  - Contains <div id="root"></div>
  - React app mounts here

File: favicon.ico
  - Browser tab icon (16x16 or 32x32 pixels)
  - Shows next to page title

File: manifest.json
  - PWA (Progressive Web App) configuration
  - Defines app name, icons, theme colors
  - Allows "Add to Home Screen" on mobile

File: robots.txt
  - SEO instructions for web crawlers
  - Tells search engines which pages to index

File: logo files (logo192.png, logo512.png)
  - App icons for different devices/screen sizes
  - Used in PWA manifest

â˜… KEY POINT: Files in public/ are NOT processed by webpack/vite
              They are copied directly to build folder
              Reference them with absolute paths: /logo.png


DIRECTORY: src/
---------------
Purpose: ALL source code that gets processed and bundled

FILE: index.js / main.jsx (ENTRY POINT)
----------------------------------------
The FIRST file executed when app starts

Example Code:
  import React from 'react'
  import ReactDOM from 'react-dom/client'
  import App from './App'
  import './index.css'

  ReactDOM.createRoot(document.getElementById('root')).render(
    <React.StrictMode>
      <App />
    </React.StrictMode>
  )

What it does:
  1. Imports React and ReactDOM
  2. Finds the DOM element with id="root"
  3. Creates a React root
  4. Renders the <App /> component into it
  5. Imports global CSS


FILE: App.js / App.jsx (ROOT COMPONENT)
----------------------------------------
The main application component - root of component tree

Example Code:
  function App() {
    return (
      <div className="App">
        <header>
          <h1>My React App</h1>
        </header>
        {/* Other components go here */}
      </div>
    )
  }
  export default App

What it does:
  - Root component of your application
  - All other components branch from here
  - Contains main routing and layout


FILE: index.css
---------------
Purpose:
  - GLOBAL styles (applies to entire app)
  - Resets, typography, CSS variables
  - Imported in index.js/main.jsx

Example:
  body { margin: 0; font-family: Arial, sans-serif; }
  * { box-sizing: border-box; }


FILE: App.css
-------------
Purpose:
  - Component-specific styles for App component
  - Imported in App.js/App.jsx
  - Scoped to App component (by convention, not technically)


CRA-SPECIFIC FILES
------------------

FILE: setupTests.js
Purpose:
  - Configures Jest testing environment
  - Imports @testing-library/jest-dom
  - Adds custom matchers (toBeInTheDocument, etc.)

FILE: App.test.js
Purpose:
  - Example test file
  - Shows how to write component tests
  - Uses React Testing Library

Example:
  test('renders learn react link', () => {
    render(<App />);
    const linkElement = screen.getByText(/learn react/i);
    expect(linkElement).toBeInTheDocument();
  });

FILE: reportWebVitals.js
Purpose:
  - Measures performance metrics
  - Tracks: FCP, LCP, CLS, FID, TTFB
  - Optional analytics integration

FILE: logo.svg
Purpose:
  - React logo used in default App component
  - Can be deleted/replaced


VITE-SPECIFIC
-------------

DIRECTORY: src/assets/
Purpose:
  - Images, fonts, and other assets
  - These ARE processed by Vite
  - Can be imported in JavaScript/JSX
  
Example:
  import logo from './assets/logo.png'
  <img src={logo} alt="Logo" />

6. WHY FILE STRUCTURE MATTERS
================================================================================

1. SEPARATION OF CONCERNS
   - Keep static assets (public/) separate from source code (src/)
   - Different treatment during build process
   - Clear organization

2. BUILD OPTIMIZATION
   - Files in src/ get minified, transpiled, and bundled
   - Tree-shaking removes unused code
   - Code splitting and lazy loading possible

3. MAINTAINABILITY
   - Organized structure = easier navigation
   - New team members can understand quickly
   - Scales better as project grows

4. PERFORMANCE
   - Proper asset management enables optimization
   - Lazy loading of routes/components
   - Efficient caching strategies

5. TEAM COLLABORATION
   - Standard structure helps everyone
   - Clear conventions (where to put what)
   - Reduces decision fatigue

6. DEVELOPER EXPERIENCE
   - Fast dev server (especially Vite)
   - Hot module replacement
   - Quick feedback loop

================================================================================
BEST PRACTICES
================================================================================

1. NEVER commit node_modules/ to Git
2. Use .env files for environment variables (API keys, URLs)
3. Keep components small and focused (single responsibility)
4. Use meaningful file/folder names
5. Group related files together (component + styles + tests)
6. Keep public/ for static assets that don't need processing
7. Import assets in src/ when they need optimization
8. Use absolute imports with path aliases (@/components)
9. Follow consistent naming conventions (PascalCase for components)
10. Document complex logic and non-obvious decisions
